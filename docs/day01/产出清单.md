## 1. Day01 核心目标

第一天只要求亲手走通这一条最短主链路：

`createApp(App)`  
`-> app.mount('#app')`  
`-> 创建根 vnode`  
`-> 进入 renderer`  
`-> 挂载组件实例`  
`-> 执行 setup`  
`-> 执行渲染函数`  
`-> vnode patch 成真实 DOM`

只要这条链路能通过断点走通，并且能用自己的话讲清楚，Day01 就算完成。

`packages/runtime-dom/src/index.ts`
- `createApp` 内部先通过 `ensureRenderer()` 拿到 renderer
- `ensureRenderer -> createRenderer -> baseCreateRenderer`
- `baseCreateRenderer` 返回的是 renderer 能力对象，里面包含 `render`、`hydrate`、`createApp`

`packages/runtime-core/src/apiCreateApp.ts`
- renderer 上的 `createApp` 来自 `createAppAPI`
- `createAppAPI` 返回闭包 `createApp`，执行后得到 `app`

`packages/runtime-dom/src/index.ts`
- `runtime-dom` 对 `app.mount` 又包了一层，专门处理 DOM 平台逻辑：
  - 把 `containerOrSelector` 解析成真实 `container`
  - 当根组件没有 `render/template` 时，才把 `container.innerHTML` 作为 `component.template`
  - 挂载前清空 `container`
  - 调用原本的 `mount`
  - 挂载后移除 `v-cloak`，设置 `data-v-app`

`packages/runtime-core/src/apiCreateApp.ts`
- 原始 `mount` 先判断应用是否已经挂载
- 如果没有挂载，则执行 `createVNode(rootComponent, rootProps)` 创建根 vnode
- 然后调用 `render(vnode, rootContainer, namespace)`

`packages/runtime-core/src/vnode.ts`
- `createVNode -> _createVNode`
- 在这里根据 `type` 得到初始 `shapeFlag`
- 再通过 `createBaseVNode` 创建 vnode

`packages/runtime-core/src/renderer.ts`
- `render` 是根渲染入口
- `vnode !== null` 时调用 `patch`
- `patch` 统一处理首次挂载和后续更新
- 在 `patch` 里根据 `shapeFlag` 分流：
  - 组件 vnode -> `processComponent`
  - 元素 vnode -> `processElement`

首次挂载组件链路：
- `processComponent`
- `mountComponent`
- `createComponentInstance`
- `setupComponent`
- `setupStatefulComponent`
- 执行 `setup()`
- `setupRenderEffect`
- `componentUpdateFn`
- `renderComponentRoot`
- 得到 `subTree`
- `patch(null, subTree, ...)`
- 如果 `subTree` 是元素 vnode，则进入 `processElement -> mountElement`

点击按钮后的组件更新链路：
- 响应式数据变化
- 触发组件更新副作用的 scheduler
- `queueJob`
- `queueFlush`
- `flushJobs`
- `instance.update`
- 再次进入 `componentUpdateFn`
- `renderComponentRoot`
- 得到新的 `nextTree`
- `patch(prevTree, nextTree, ...)`
- 如果前后还是同一种元素，一般走 `processElement -> patchElement`
- 只有首次挂载或节点类型变化时才会走 `mountElement`

### 2.3 一份状态更新链路总结

除了首次挂载，还要补一次“为什么点击按钮会更新”的链路。

至少要覆盖：

- `bump()` 或按钮点击后，为什么不是立刻同步重渲染  
  因为需要通过 scheduler 进入队列
- `scheduler.ts` 里的 `queueJob`、`queueFlush`、`flushJobs`、`nextTick`
- 组件更新时什么时候进入 `instance.update`
- 第二次进入 `patch` 时，为什么会变成新旧 vnode 对比
- 元素更新时为什么会走 `patchElement`

可直接整理成下面这段说明：

```text
`bump()` 或按钮点击后，不会立刻同步重渲染，因为组件渲染对应的不是“直接执行”的逻辑，
而是一个带 scheduler 的 ReactiveEffect。

当响应式数据变化时，会触发这个 effect；但由于 effect 配置了
`effect.scheduler = () => queueJob(job)`，
所以这次触发不会立刻重新执行组件 render，而是先把组件更新任务放进调度队列。

`queueJob(job)` 的作用是：
1. 把当前组件的更新 job 放进队列
2. 对同一个 job 做去重，避免同一轮重复入队
3. 调用 `queueFlush()` 安排一次异步刷新

`queueFlush()` 不会同步执行更新，而是通过：
`Promise.resolve().then(flushJobs)`
把这轮更新放进微任务队列。

等当前这轮同步代码执行完后，才会进入 `flushJobs()`：
1. 按顺序取出队列中的 job
2. 执行组件更新
3. 最终重新进入组件的 render
4. 再走 `patch(prevTree, nextTree, ...)`

所以“不是立刻同步重渲染”的更准确说法是：
状态变化后，不是直接重新 render，
而是先通过 scheduler 进入队列，
再由微任务中的 `flushJobs` 统一执行。

这样做的目的主要有三个：
1. 批量更新：同一轮里多次改状态，尽量合并成一次更新
2. 去重：避免同一个组件重复执行多次更新
3. 保证顺序：父组件通常先于子组件更新
```

要求：

- 能说明“更新时还是 patch，只是这次 `n1` 和 `n2` 同时存在”
- 不要求展开 keyed diff 细节

### 2.5 一份关键概念解释

Day01 至少要能用自己的话解释这些概念：

- `__DEV__` 是什么  
  编译期开发标记。开发构建时通常为 `true`，生产构建时通常为 `false`，Vue 用它包住 warning、调试信息、性能打点等只在开发期需要的逻辑。
- `container.nodeType === 1` 意味着什么  
  表示挂载容器是一个 DOM 元素节点（`Element`），不是“组件类型的节点”。Vue 在这里是为了判断这个容器能不能读 `innerHTML`、清空文本内容、移除属性等。
- 为什么挂载后要移除 `v-cloak`、设置 `data-v-app`  
  `v-cloak` 常用于在 Vue 接管前先把模板隐藏起来，挂载完成后移除它，内容就可以正常显示。  
  `data-v-app` 是一个标记，表示这个容器已经被 Vue 应用接管，不是“让组件显示出来”的关键逻辑。
- `runtime-dom` 为什么要重写一层 `mount`  
  因为 `runtime-core` 里的 `mount` 是平台无关的通用逻辑；`runtime-dom` 需要补一层浏览器 DOM 平台特有处理，比如：
  - 把选择器解析成真实容器
  - 在某些场景下把 `container.innerHTML` 当作模板来源
  - 挂载前清空容器内容
  - 挂载后处理 `v-cloak` 和 `data-v-app`
- `vnode` 的 6 个关键字段
  - `type`：节点类型。元素时一般是标签名字符串，组件时一般是组件对象或函数。
  - `props`：这个 vnode 的属性集合。对元素来说是 attrs / events 等，对组件来说包含传入的 props。
  - `children`：子内容，不一定只是“子节点数组”，也可能是文本、数组、slots 对象。
  - `shapeFlag`：位运算标记，记录当前 vnode 以及 children 的形态。
  - `el`：这个 vnode 最终对应的真实宿主节点引用。
  - `component`：如果当前 vnode 是组件 vnode，这里会挂载对应的组件实例。
- `shapeFlag` 的作用是什么  
  用位运算快速判断当前 vnode 是元素、组件、Teleport、Suspense，或者它的 children 是文本、数组还是 slots。`patch` 阶段会根据它做分流，比如走 `processElement` 还是 `processComponent`。
- `render` 在做什么  
  它是根渲染入口。接收根 vnode 和容器后，统一走 `patch`：首次挂载时是“旧 vnode 不存在”的挂载流程；后续更新时是“新旧 vnode 对比”的更新流程；传 `null` 时则执行卸载。
- `mountComponent` 在做什么  
  不只是“挂载组件”四个字。它至少做三件事：
  - 创建组件实例，并挂到组件 vnode 的 `component` 字段上
  - 调用 `setupComponent`，完成 props、slots、`setup()`、render 准备
  - 调用 `setupRenderEffect`，建立组件渲染副作用，正式进入首屏渲染

### 2.6 一份组件实例认知笔记

至少回答这些问题：

- 组件实例里大概存了什么 
  组件实例里会保存当前组件对应的 `vnode`、组件类型、父组件、`props`、`attrs`、`slots`、`refs`、`setupState`、渲染上下文 `ctx`、代理对象 `proxy`、渲染函数 `render`、渲染子树 `subTree`、更新函数 `update`、挂载状态等信息。
- `setup()` 在哪里被调用
  调用链路是 `mountComponent -> setupComponent -> setupStatefulComponent`。真正执行 `setup()` 的位置在 `runtime-core/src/component.ts` 的 `setupStatefulComponent` 里。
- `setup()` 返回函数和返回对象分别怎么处理
  当返回值是函数时，这个函数会被当作组件的渲染函数，赋值给 `instance.render`。  
  当返回值是对象时，这个对象会经过 `proxyRefs` 处理后赋值给 `instance.setupState`。
- 渲染函数最后挂到了哪里
  最终挂到组件实例的 `instance.render` 字段上。  
  如果 `setup()` 直接返回函数，就在 `handleSetupResult` 里挂上去；如果没有返回 render，则会在 `finishComponentSetup` 里把组件自己的 `render` 或模板编译结果挂到 `instance.render`。

### 2.7 一份验收问答

Day01 最终至少要能独立回答这些问题：

1. `createApp(App).mount('#app')` 最终是谁调用了 `render`？
   `runtime-dom` 重写过的 `app.mount` 在处理完 DOM 容器后，会调用 `runtime-core` 的原始 `mount`；原始 `mount` 在创建根 vnode 后执行 `render(vnode, rootContainer, namespace)`。
2. 根组件是在哪一步变成 vnode 的？
   根组件是在 `app.mount` 里执行 `createVNode(rootComponent, rootProps)` 时变成根 vnode 的。
3. 组件实例是在哪一步创建的？
   调用链路是 `render -> patch -> processComponent -> mountComponent`，然后在 `mountComponent` 里通过 `createComponentInstance` 创建组件实例。
4. `setup()` 是在哪一步调用的？
   调用链路是 `mountComponent -> setupComponent -> setupStatefulComponent`，真正执行 `setup()` 的位置在 `setupStatefulComponent` 里。
5. 首次挂载时，组件 vnode 和元素 vnode 分别走哪个分支？
   在 `patch` 里，组件 vnode 先走 `processComponent`，元素 vnode 先走 `processElement`；首次挂载时再分别继续进入 `mountComponent` 和 `mountElement`。
6. 状态变化后，为什么不是立刻同步重新渲染？
   因为组件更新不是直接同步执行 render，而是先通过 scheduler 把更新任务放进队列：`queueJob` 负责入队和去重，`queueFlush` 把刷新安排到微任务，最后由 `flushJobs` 统一执行更新。
7. 第二次更新时，为什么还是走 `patch`？
   因为 `patch` 是 Vue 统一的渲染入口，首次挂载和后续更新都走它。更新时会生成新的 vnode，然后把旧 vnode 和新 vnode 一起传给 `patch` 做对比更新。
