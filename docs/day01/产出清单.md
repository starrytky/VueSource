## 1. Day01 核心目标

第一天只要求亲手走通这一条最短主链路：

`createApp(App)`  
`-> app.mount('#app')`  
`-> 创建根 vnode`  
`-> 进入 renderer`  
`-> 挂载组件实例`  
`-> 执行 setup`  
`-> 执行渲染函数`  
`-> vnode patch 成真实 DOM`

只要这条链路能通过断点走通，并且能用自己的话讲清楚，Day01 就算完成。

## 2. 必须产出

以下内容属于 Day01 的必交付内容。

### 2.1 一份主链路笔记

至少要整理出下面这条调用链：

```text
业务代码
  -> runtime-dom createApp / mount
  -> runtime-core createAppAPI
  -> createVNode
  -> renderer.render
  -> renderer.patch
  -> processComponent
  -> mountComponent
  -> setupComponent
  -> setupRenderEffect
  -> renderComponentRoot
  -> patch subtree
  -> processElement
  -> mountElement
```

要求：

- 说明每一步在哪个文件
- 说明每一步的职责是什么
- 区分“组件 vnode 链路”和“元素 vnode 链路”

### 2.2 一份首次挂载 10 步总结

至少要能整理出这 10 个关键事实：

1. `createApp` 最终会生成根 vnode
2. `render` 会把根 vnode 交给 `patch`
3. `patch` 发现根 vnode 是组件，于是走 `processComponent`
4. `mountComponent` 里会创建组件实例
5. `setupComponent` 里会执行 `setup`
6. `setupRenderEffect` 里会执行组件渲染
7. 渲染结果会变成子树 vnode
8. 子树 vnode 再次进入 `patch`
9. 如果子树是原生元素，就走 `processElement -> mountElement`
10. 最后真实 DOM 被插入容器

要求：

- 这 10 步要能串成一段完整叙述
- 最好能配一张调用流程图

### 2.3 一份状态更新链路总结

除了首次挂载，还要补一次“为什么点击按钮会更新”的链路。

至少要覆盖：

- `bump()` 或按钮点击后，为什么不是立刻同步重渲染
- `scheduler.ts` 里的 `queueJob`、`queueFlush`、`flushJobs`、`nextTick`
- 组件更新时什么时候进入 `instance.update`
- 第二次进入 `patch` 时，为什么会变成新旧 vnode 对比
- 元素更新时为什么会走 `patchElement`

要求：

- 能说明“更新时还是 patch，只是这次 `n1` 和 `n2` 同时存在”
- 不要求展开 keyed diff 细节

## 5. Day01 不纳入产出范围

第一天先不要把精力花在这些内容上：

- 编译器 `compiler-core` / `compiler-dom`
- `Transition`
- `Teleport`
- `Suspense`
- `KeepAlive`
- SSR / hydration
- 复杂 diff 细节

原则：

- 先走通主链路
- 先建立运行时骨架认知
- 暂时不深挖高阶分支
