## 1. Day02 核心目标

第二天的目标不是继续扩散到整个响应式系统，而是把“组件更新是怎样被调度起来的”这一条链路亲手走通。

第二天只要求走通下面这条更新主链：

`修改响应式状态`  
`-> trigger(...)`  
`-> 组件 render effect 被触发`  
`-> effect.scheduler()`  
`-> queueJob(instance.job)`  
`-> queueFlush()`  
`-> flushJobs()`  
`-> componentUpdateFn`  
`-> renderComponentRoot(instance)`  
`-> patch(prevTree, nextTree)`  
`-> pre / post watcher 与生命周期钩子插入执行`

只要这条链路能通过断点走通，并且能用自己的话讲清楚，Day02 就算完成。

`packages/reactivity/src/baseHandlers.ts`
- 响应式对象被修改时，通常从 `MutableReactiveHandler.set` / `deleteProperty` 进入
- 然后调用 `trigger(...)`

`packages/reactivity/src/dep.ts`
- `trigger(...)` 负责找到和当前 target / key 相关的依赖
- `Dep.trigger -> notify` 会继续通知订阅它的 effect

`packages/reactivity/src/effect.ts`
- 组件 render effect 最终会进入 `ReactiveEffect.trigger`
- 如果 effect 配置了 `scheduler`，这里不会直接 `runIfDirty()`，而是改走 `scheduler()`

`packages/runtime-core/src/renderer.ts`
- `setupRenderEffect` 里给组件更新 effect 绑定了 scheduler
- 关键代码是：
  - `instance.job = effect.runIfDirty.bind(effect)`
  - `job.id = instance.uid`
  - `effect.scheduler = () => queueJob(job)`
- 更新真正执行时会进入 `componentUpdateFn`
- `componentUpdateFn` 里会重新 `render`，然后 `patch(prevTree, nextTree)`

`packages/runtime-core/src/scheduler.ts`
- `queueJob` 负责入队、去重、按 id 插队
- `queueFlush` 负责用微任务安排一次 flush
- `flushJobs` 负责顺序执行队列中的 job
- `flushPreFlushCbs` / `flushPostFlushCbs` 负责在主更新流程前后插入执行 watcher / 生命周期类回调

`packages/runtime-core/src/apiWatch.ts`
- `doWatch` 决定 watcher 走 `sync` / `pre` / `post` 哪条调度路径

第二天最核心的结论只有一句话：

```text
组件更新不是“状态一变就立刻重新 render”，
而是“状态变化 -> 触发 effect -> scheduler 入队 -> 微任务 flush -> render + patch”。
```

### 2.2 一份“更新 8 步”总结

至少要能整理出这 8 个关键事实：

1. 修改响应式状态时，会从 `set` / `deleteProperty` 进入 `trigger`
2. 组件 render effect 会被依赖通知到
3. effect 不一定立刻重跑，组件 effect 通常会交给 scheduler
4. scheduler 会把组件 job 放进队列，并给同一个 job 去重
5. `queueFlush()` 不是同步执行，而是通过微任务安排 `flushJobs()`
6. `flushJobs()` 真正执行组件更新 job
7. 组件更新 job 会重新执行 `renderComponentRoot(instance)`
8. 最终通过 `patch(prevTree, nextTree)` 完成一次更新

要求：

- 这 8 步要能串成一段完整叙述
当业务代码修改响应式状态（如 count.value++）时：

触发入口：count.value 的 Proxy set handler（MutableReactiveHandler.set）被调用，随即进入 trigger(target, TriggerOpTypes.SET, key)

依赖通知：trigger() 在 dep.ts 中找到所有依赖该属性的订阅者（ReactiveEffect），调用 dep.notify() 逐个通知

调度权转移：组件的 render effect 被通知后，不走同步 runIfDirty()，而是走 effect.scheduler()。在 renderer.ts 的 setupRenderEffect 中，组件 effect 的 scheduler 被设置为 queueJob(instance.job)

入队与去重：queueJob() 检查 job 是否有 QUEUED 标记，有则跳过，无则插入队列并标记，确保同一组件的更新不会重复入队

微任务安排：queueFlush() 不直接执行更新，而是通过 Promise.resolve().then(flushJobs) 安排一个微任务，保证本轮同步代码执行完后再批量更新

批量执行：微任务触发时 flushJobs() 真正开始执行，按 id 顺序依次调用 componentUpdateFn

重新渲染：componentUpdateFn 调用 renderComponentRoot(instance) 生成新 vnode 树

DOM 更新：patch(prevTree, nextTree) 对比新旧 vnode 树，将差异同步到真实 DOM

### 2.3 一份这 5 个问题的答案

第二天最少要把下面这 5 个问题回答到“能口述源码依据”的程度：

1. 同一个 tick 内多次改状态，为什么通常只会触发一次组件更新？
2. 调度器怎样避免同一个组件重复入队？
3. 为什么组件更新顺序通常是父组件先、子组件后？
4. `watch` 的 `flush: 'pre' | 'post' | 'sync'` 到底分别插在哪里？
5. 组件更新时，什么时候只是更新 `props/slots`，什么时候会真正进入重新 `render + patch`？

每个问题至少要包含两部分：

- 结论
- 源码位置

建议最低整理成下面这种粒度：

```text
问题：同一个 tick 内多次改状态，为什么通常只更新一次？
结论：因为多次触发最后会落到同一个 `instance.job`，而 `queueJob` 会通过 `QUEUED` 标记去重。
详解：
      1.instance.update 就是 instance.job
        - 在 setupRenderEffect 中，会创建一个 ReactiveEffect 实例
        - 这个 effect.run 被包装成 job，并赋值给 instance.update
        - 同一个组件实例的 instance.update 始终是同一个函数引用

      2.响应式触发流程
        - 修改响应式数据 -> trigger() -> triggerEffects()
        - 遍历所有订阅 effect，调用 effect.scheduler
        - 组件的 effect.scheduler 就是 queueJob(instance.update)
      
      3.多次修改为什么落到同一个job
        例如：
        state.count++ // 第一次修改
        state.count++ // 第二次修改
        state.count++ // 第三次修改
        每次修改都会：
        - 触发 trigger -> 调用 effect.scheduler
        - 执行 queueJob(instance.update)
        - 但传入的都是同一个 instance.update 函数引用
      
      4.queueJob 的去重机制
        - queueJob 内部维护了一个 queue 数组
        - 入队前检查： if (job.flags & SchedulerJobFlags.QUEUED) return
        - 如果这个 job 已经在队列中（已标记 QUEUE），执行跳过
        - 所以第二次、第三次 queueJob(instance.update) 都会被忽略
      
      5.最终只执行一次
        - 微任务执行时，flushJobs 遍历 queue
        - 每个 job 只在队列中出现一次
        - 所以 instance.update 只会被调用一次
        - 这一次调用会读取最新的响应式数据（count 已经 +3）

      源码关键位置：
        - `renderer.ts` -> setupRenderEffect: 
          const effect = new ReactiveEffect(componentUpdateFn)
          const update = (instance.update = effect.run.bind(effect))
          update.i = instance
          
        - `reactivity/effect.ts` -> triggerEffect:
          if (effect.scheduler) {
            effect.scheduler()
          }
          
        - `renderer.ts` -> setupRenderEffect 中设置 scheduler:
          effect.scheduler = () => queueJob(update)
          
        - `scheduler.ts` -> queueJob:
          if (job.flags & SchedulerJobFlags.QUEUED) return
          job.flags |= SchedulerJobFlags.QUEUED
          queue.push(job)
        

源码位置：
- `renderer.ts` -> `setupRenderEffect`
- `scheduler.ts` -> `queueJob`
```

```text
问题 2：调度器怎样避免同一个组件重复入队?
结论：通过 job.flags 的 QUEUED 标记位来去重。入队前检查 flags & QUEUED，如果已标记则跳过；入队时设置 QUEUED 标记，执行后清除标记。
源码位置：
- `scheduler.ts` -> `queueJob` 函数中的去重逻辑
- `scheduler.ts` -> `SchedulerJobFlags.QUEUED` 标记位定义
- `renderer.ts` -> `setupRenderEffect` 创建的 job 会被复用
```


```text
问题 3：为什么组件更新顺序通常是父组件先、子组件后？
结论：因为组件创建时会递增分配 uid，父组件的 uid 小于子组件。调度器在 flushJobs 时会对队列按 job.id 排序，而 job.id 通常就是 instance.uid，所以父组件先执行。
源码位置：
- `scheduler.ts` -> `flushJobs` 中的 queue.sort(comparator)
- `scheduler.ts` -> comparator 比较函数按 id 升序排序
- `component.ts` -> 组件实例创建时分配递增的 uid
```


```text
问题 4：`watch` 的 `flush: 'pre' | 'post' | 'sync'` 到底分别插在哪里？
结论：
- `sync`：立即同步执行，不入队
- `pre`：通过 queueJob 插入主队列，在组件更新前执行
- `post`：通过 queuePostFlushCb 插入 postFlushCbs 队列，在组件更新后执行
源码位置：
- `apiWatch.ts` -> `doWatch` 函数中根据 flush 选项决定调度方式
- `scheduler.ts` -> `queueJob` (pre) 和 `queuePostFlushCb` (post)
- `scheduler.ts` -> `flushJobs` 先执行主队列再执行 postFlushCbs
```


```text
问题 5：组件更新时，什么时候只是更新 `props/slots`，什么时候会真正进入重新 `render + patch`？
结论：在 updateComponent 中会先调用 shouldUpdateComponent 判断。如果 props/slots/directives 等没有实质变化，只会调用 updateHOCHostEl 更新引用，不会触发重新渲染；否则会设置 next 并调度 instance.update 进入完整的 render + patch 流程。
源码位置：
- `renderer.ts` -> `updateComponent` 函数
- `componentRenderUtils.ts` -> `shouldUpdateComponent` 判断逻辑
- `renderer.ts` -> `setupRenderEffect` 中的 componentUpdateFn 处理 next
```

可直接整理成下面这段说明：

```text
组件更新之所以会合并，是因为响应式触发最终并不是直接调用 render，
而是调用 `queueJob(instance.job)`。

`queueJob` 做了三件事：
1. 检查当前 job 是否已经带有 `QUEUED` 标记
2. 没有就按 id 插入调度队列
3. 调用 `queueFlush()` 安排一次微任务刷新

`queueFlush()` 只负责保证本轮会有一次 `flushJobs()`，
不会每次状态变化都立刻同步跑更新。

真正执行时，`flushJobs()` 会：
1. 顺序取出主队列中的 job
2. 执行组件更新
3. 在结尾统一跑 `flushPostFlushCbs()`

如果在组件更新前需要插入 pre watcher，
则会通过 `flushPreFlushCbs(instance)` 先执行。
```

要求：

- 能把“主队列”和“pre/post 队列”区分开
- 能说清楚 `getId(job)` 对更新顺序的影响

### 2.5 一份 watch flush 时机总结

至少要把 `packages/runtime-core/src/apiWatch.ts` 的 `doWatch` 整理成一份对照表。

最低要覆盖：

- `flush: 'sync'`
  - 不走自定义 scheduler
  - 依赖一触发就直接执行
- `flush: 'pre'`
  - 默认值
  - 首次运行同步执行
  - 后续走 `queueJob(job)`
  - 会打上 `PRE` 标记
- `flush: 'post'`
  - 走 `queuePostRenderEffect(job, suspense)`
  - 最终落到 `queuePostFlushCb(...)`

建议最少整理出这张表：

| flush 选项 | 调度方式 | 执行时机 | 适合观察什么 |
| --- | --- | --- | --- |
| `sync` | 直接执行 job | 依赖触发当下 | 是否要立即同步响应 |
| `pre` | `queueJob(job)` + `PRE` 标记 | 本轮组件 `render` 之前 | 更新前状态、更新前 DOM |
| `post` | `queuePostRenderEffect(...)` | 本轮 `patch` 之后 | 更新后 DOM |

要求：

- 能解释为什么 `pre` 常常看到更新前 DOM
- 能解释为什么 `post` 更适合读 DOM

### 2.6 一份“组件更新来源”对照笔记

第二天一定要区分两类更新来源：

#### 组件自身状态变化

- `instance.next` 通常是 `null`
- 直接由组件自己的响应式依赖触发更新 effect

更新路径：

```text
响应式状态变化
  -> queueJob(instance.job)
  -> flushJobs
  -> componentUpdateFn
  -> next 为空
  -> next = vnode
  -> renderComponentRoot
  -> patch(prevTree, nextTree)
```

#### 父组件传入新 vnode / 新 props

表现：

- `updateComponent(n1, n2, optimized)` 先执行
- `instance.next = n2`
- 进入 `componentUpdateFn` 时，`next` 不为空

```text
父组件 patch 子组件 vnode
  -> updateComponent
  -> instance.next = n2
  -> instance.update()
  -> componentUpdateFn
  -> updateComponentPreRender(instance, next, optimized)
  -> updateProps / updateSlots
  -> flushPreFlushCbs(instance)
  -> renderComponentRoot
  -> patch(prevTree, nextTree)
```

要求：

- 明确区分“next 为空”和“next 不为空”两种情况
- 知道 `updateComponentPreRender` 的职责是先同步 `props / slots`
