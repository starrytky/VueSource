## 1. Day03 核心目标

第三天的目标不是继续扩散到 diff、DOM patch 或编译器，而是把“谁触发了 `effect.scheduler`”这条链路在 `reactivity` 里亲手走通。

第三天只要求走通下面这条主链：

`ReactiveEffect.run()`  
`-> activeSub = 当前 effect`  
`-> Proxy.get / ref.value get`  
`-> track(...)`  
`-> targetMap / dep / link 建立依赖关系`  
`-> Proxy.set / ref.value set`  
`-> trigger(...)`  
`-> dep.notify()`  
`-> batch(...)`  
`-> ReactiveEffect.trigger()`  
`-> effect.scheduler() 或 runIfDirty()`  
`-> 回到第二天的 scheduler / 组件更新链路`

只要这条链路能通过断点走通，并且能用自己的话讲清楚，Day03 就算完成。

`packages/runtime-core/src/renderer.ts`
- `setupRenderEffect` 会为组件创建 render effect
- 关键代码是：
  - `const effect = new ReactiveEffect(componentUpdateFn)`
  - `instance.job = effect.runIfDirty.bind(effect)`
  - `effect.scheduler = () => queueJob(job)`
- 第三天在这里的任务不是继续研究 scheduler，而是确认“组件 effect 确实来自 `ReactiveEffect`”

`packages/reactivity/src/effect.ts`
- `ReactiveEffect.run` 负责把当前 effect 放进 `activeSub`
- `ReactiveEffect.notify` / `trigger` 负责 effect 被依赖通知后如何继续触发
- `batch` / `startBatch` / `endBatch` 负责把 subscriber 统一批处理
- `refreshComputed` 负责 computed 的惰性刷新和缓存判断

`packages/reactivity/src/baseHandlers.ts`
- `BaseReactiveHandler.get` 负责普通对象读取时的依赖收集
- `MutableReactiveHandler.set` / `deleteProperty` 负责对象写入后的触发
- `has` / `ownKeys` 负责“存在性”和“遍历结构”的依赖收集

`packages/reactivity/src/dep.ts`
- `targetMap` 是对象属性依赖的大总表
- `Dep` 是某个 key 的订阅中心
- `Link` 是 `dep <-> subscriber` 的双向连接
- `track(...)` 负责建立依赖
- `trigger(...)` 负责找到受影响的 dep 并继续通知

`packages/reactivity/src/ref.ts`
- `RefImpl` 自己维护 `dep`
- `get value` 走 `dep.track()`
- `set value` 走 `dep.trigger()`

`packages/reactivity/src/computed.ts`
- `ComputedRefImpl` 既有自己的 `dep`，自己也是 `Subscriber`
- 依赖变更时先 `DIRTY`
- 真正读取 `computed.value` 时才 `refreshComputed(...)`

第三天最核心的结论只有一句话：

```text
第二天看到的 `effect.scheduler()` 不是凭空被调用的，
它的上游是 `reactivity` 里“读取时收集依赖、写入时触发依赖”的整套机制。
```

### 2.3 一份“依赖收集 6 步”总结

至少要能整理出这 6 个关键事实：

1. 组件 render effect 运行时，会在 `ReactiveEffect.run()` 里把自己放进 `activeSub`
2. 普通对象读取会通过 Proxy `get` 进入 `track(target, key)`
3. `track(...)` 会在 `targetMap` 中找到或创建当前 key 对应的 `dep`
4. `dep.track()` 会创建或复用 `Link`
5. `Link` 会同时挂到 subscriber 和 dep 两边
6. 这样下次写入同一个 key 时，就能反向找到依赖它的 subscriber

要求：

- 这 6 步要能串成一段完整叙述
- 最好能配一张“依赖收集图”

### 2.4 一份“触发更新 7 步”总结

至少要能整理出这 9 个关键事实：

1. 写操作会从 `set` / `deleteProperty` / `ref.value set` 进入触发逻辑
2. `trigger(...)` 会从 `targetMap` 或 `ref.dep` 找到受影响的 dep
3. `dep.trigger()` 会递增版本并调用 `notify()`
4. `notify()` 会先 `startBatch()`，再遍历订阅者调用 `sub.notify()`
5. `ReactiveEffect.notify()` 会调用 `batch(this)`，把自己放进批处理链
6. `batch()` 会区分 computed subscriber 和普通 effect，分别放进 `batchedComputed` 和 `batchedSub` 两条链
7. `endBatch()` 先处理 `batchedComputed`（重置 computed 标志），再处理 `batchedSub`（调用 `ReactiveEffect.trigger()`）
8. `ReactiveEffect.trigger()` 会根据有没有 `scheduler` 决定后续路径
9. 组件 render effect 如果带 `scheduler`，最终就会回到 `queueJob(instance.job)`

要求：

- 这 9 步要能串成一段完整叙述
- 要能解释”为什么不是写操作一发生就直接重新 render”
- 要能解释 batch 的两层队列（`batchedComputed` / `batchedSub`）为什么需要分开处理

4. `computed` 为什么默认是惰性的，依赖变了为什么通常不是立刻重新计算？
5. 条件分支切换后，为什么旧分支上的依赖会被清掉？

每个问题至少要包含两部分：

```text

结论：因为依赖变更时通常只是把 computed 标记为 DIRTY，
真正读取 `computed.value` 时才会进入 `refreshComputed()` 判断是否需要重新求值。
此外，computed 在失去所有 subscriber 时会通过 removeSub 退订自身依赖，
获得首个 subscriber 时才通过 addSub 重新订阅，从根源上避免无谓计算。
源码位置：
- `computed.ts` -> `ComputedRefImpl.notify`
- `computed.ts` -> `get value`
- `effect.ts` -> `refreshComputed`
- `dep.ts` -> `addSub`
- `effect.ts` -> `removeSub`
```

### 2.6 一份 `targetMap / Dep / Link` 认知笔记

至少要把 `packages/reactivity/src/dep.ts` 里的这些点整理出来：

- `targetMap`
- `KeyToDepMap`
- `Dep`
- `Link`
- `track`
- `Dep.track`
- `trigger`
- `Dep.trigger`
- `Dep.notify`

至少要解释清楚下面这些事实：

- `targetMap` 为什么是 `WeakMap<object, Map<key, Dep>>`
- 为什么 Vue 现在不是简单地用 `Set<effect>`
- `Link` 为什么要同时挂在 dep 和 subscriber 两边
- 为什么这种结构更适合做依赖清理和复用

可直接整理成下面这段说明：

```text
依赖收集时，Vue 并不是简单地把 “当前 key 对应哪些 effect”
丢进一个 `Set` 就结束了，而是构建了一个双向结构：

1. `targetMap` 先根据 target 找到 `depsMap`
2. `depsMap` 再根据 key 找到对应的 `dep`
3. `dep.track()` 为当前 active subscriber 创建或复用 `Link`
4. `Link` 一边挂在 subscriber 的依赖链上，一边挂在 dep 的订阅链上

这样做的好处是：
1. 写入时能从 dep 很快找到订阅者
2. effect 重跑后能从 subscriber 这边回收无用依赖
3. 老依赖可以复用，而不是每轮都暴力重建
```

要求：

- 能把”对象 -> key -> dep”和”dep <-> subscriber”两层结构区分开
- 能说明 `Link.version` 在清理和脏检查中的作用（清理时置 -1 标记失效；`isDirty()` 中 `link.dep.version !== link.version` 判断依赖是否变化过）

### 2.7 一份 `reactive / ref / computed` 对照笔记

第三天一定要把这三类响应式载体区分开。

#### `reactive`

表现：

- 基于 `Proxy`
- 依赖收集入口在 `BaseReactiveHandler.get`
- 触发入口在 `MutableReactiveHandler.set / deleteProperty`

更新路径：

```text
reactive 对象属性读取
  -> Proxy.get
  -> track(target, key)

reactive 对象属性写入
  -> Proxy.set / delete
  -> trigger(target, key)
```

#### `ref`

表现：

- 不是 Proxy，而是一个带 `dep` 的对象实例
- 依赖收集入口在 `get value`
- 触发入口在 `set value`

更新路径：

```text
ref.value get
  -> ref.dep.track()

ref.value set
  -> ref.dep.trigger()
```

#### `computed`

表现：

- 自己有 `dep`
- 自己也是 `Subscriber`
- 默认惰性求值

更新路径：

```text
computed 依赖变化
  -> ComputedRefImpl.notify()
  -> 标记 DIRTY
  -> 下次读取 computed.value
  -> refreshComputed()
  -> 需要时才重新求值
```

要求：

- 明确区分“谁在 track、谁在 trigger、依赖存在哪”
- 知道 `computed` 和普通 `ref` 最大的区别是“缓存 + 脏标记 + 惰性刷新”

### 2.8 一份依赖清理机制笔记

第三天一定要把“分支依赖为什么会自动失效”整理清楚。

至少要覆盖：

- `prepareDeps`
- `cleanupDeps`
- `Link.version`
- `isDirty`

建议至少整理成下面这种逻辑：

```text
effect 每次重新运行前，Vue 不会直接把旧依赖全删掉重建，
而是先把旧 link 的 version 统一置为 -1。

本轮运行时，真正再次访问到的 dep，
会在 `Dep.track()` 里把对应 link 的 version 同步回来。

最后 `cleanupDeps()` 会把仍然是 -1 的 link 清掉，
表示这些依赖在本轮没有再被访问，已经失效。

此外，`Link.version` 还有一个关键用途：
`isDirty()` 会遍历 subscriber 的依赖链，
如果发现 `link.dep.version !== link.version`，说明依赖被触发过（版本已递增），
就判定当前 subscriber 是"脏的"，需要重新运行。
```

要求：

- 能解释为什么分支切换后旧依赖会被清掉
- 能解释为什么这个机制和 computed 的脏判断是相通的
- 能说明 `Link.version` 同时服务于"依赖清理"和"脏检查"两个场景

### 2.11 一份关键概念解释

第三天至少要能用自己的话解释这些概念：

- `activeSub`
  - 表示“当前正在运行、因此可能被收集依赖的 subscriber”
- `shouldTrack`
  - 表示当前这次读取是否允许参与依赖收集
- `targetMap`
  - 表示整个响应式对象属性依赖的大总表
- `Dep`
  - 表示某一个具体依赖点的订阅中心
- `Link`
  - 表示某个 dep 和某个 subscriber 之间的一条双向连接
- `Subscriber`
  - 是更底层的抽象，组件 render effect、watcher、computed 都可以是 subscriber
- `DIRTY`
  - 表示 computed 或 subscriber 当前缓存可能已失效
- `globalVersion`
  - 表示“自上次以来全局有没有发生过响应式变化”的快速判断标记

要求：

- 不只是背定义
- 要能放回依赖收集和触发链里解释它们分别出现在哪里
