# Day05 产出清单

> 根据 `VueSource/docs/vue源码阅读第五天打断点.md` 整理  
> 目标：把“第五天到底要产出什么”收敛成可执行、可验收的清单

## 1. Day05 核心目标

第五天的目标不是继续停留在 `renderer` diff，而是把 `runtime-dom` 这一层的宿主实现亲手走通。

第五天只要求走通下面这条主链：

`runtime-dom/src/index.ts`  
`-> rendererOptions = nodeOps + patchProp`  
`-> createRenderer(rendererOptions)`  
`-> patchProp(...)`  
`-> class / style / attrs / events / DOM prop`

只要这条链路能通过断点走通，并且能用自己的话讲清楚，Day05 就算完成。

## 2. 必须产出

### 2.1 一份平台分层笔记

至少要整理出下面这条关系：

```text
runtime-core
  -> 平台无关 renderer

runtime-dom
  -> nodeOps + patchProp
  -> DOM 平台宿主实现
```

要求：

- 说明为什么 `runtime-core` 不直接依赖 DOM API
- 说明 `runtime-dom` 是怎样注入宿主能力的

### 2.2 一份 `patchProp` 分流总结

至少要覆盖：

- `class -> patchClass`
- `style -> patchStyle`
- `event -> patchEvent`
- `prop / attr` 分流

要求：

- 能解释为什么这些更新不能混成一套逻辑
- 能说明 `shouldSetAsProp` 的作用

### 2.3 一份事件更新总结

至少要说明：

1. `patchEvent` 不只是简单 add/remove
2. Vue 会尽量复用 invoker
3. 事件回调变化时，通常只是更新 invoker.value
4. 这样能减少原生事件反复解绑 / 重绑

### 2.4 一份这 5 个问题的答案

至少能回答：

1. `runtime-core` 为什么不直接写 DOM API？
2. `runtime-dom` 是怎样把平台能力注入 renderer 的？
3. `patchProp` 为什么还要再分流？
4. prop 和 attr 的区别是什么？
5. 为什么事件更新通常不会每次都重新绑原生监听器？

### 2.7 一份验收问答

至少要能独立回答：

1. 为什么 Vue 要把 `runtime-core` 和 `runtime-dom` 分层？
2. `patchProp` 为什么是“总分发器”而不是最终落点？
3. 什么情况下更适合走 DOM prop，什么情况下更适合走 attr？
4. 为什么事件更新更适合复用 invoker？
5. `class / style / attrs / events` 为什么适合独立模块？

## 3. 建议产出形式

推荐至少整理成下面 4 份内容：

- 一张平台分层图
- 一张 `patchProp` 分流图
- 一张事件更新图
- 一份结构化笔记：平台分层、属性更新、事件更新、验收问答

## 4. 推荐阅读顺序

1. `runtime-dom/src/index.ts`
2. `runtime-dom/src/nodeOps.ts`
3. `runtime-dom/src/patchProp.ts`

5. `modules/style.ts`

## 5. Day05 不纳入产出范围

- `compiler-core`
- `.vue` 单文件组件编译
- SSR / hydration
- 自定义元素边角分支
