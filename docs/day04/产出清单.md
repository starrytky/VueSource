# Day04 产出清单

> 根据 `VueSource/docs/vue源码阅读第四天打断点.md` 整理  
> 目标：把“第四天到底要产出什么”收敛成可执行、可验收的清单

## 1. Day04 核心目标

第四天的目标不是继续扩散到 `reactivity` 或 DOM 平台细节，而是把 `renderer` 更新元素时的 diff 主链亲手走通。

第四天只要求走通下面这条主链：

`patch(prevTree, nextTree)`  
`-> processElement(...)`  
`-> patchElement(...)`  
`-> patchChildren(...)`  
`-> patchKeyedChildren(...)`  
`-> move / unmount / mount`

只要这条链路能通过断点走通，并且能用自己的话讲清楚，Day04 就算完成。

## 2. 必须产出

### 2.1 一份元素更新主链笔记

至少要整理出下面这条调用链：

```text
组件更新
  -> patch(prevTree, nextTree)
  -> processElement
  -> patchElement
  -> patchChildren
  -> patchKeyedChildren（数组 children 场景）
```

要求：

- 说明每一步在哪个文件
- 说明每一步的职责是什么
- 说明“为什么同类型元素不是重新 mount”

### 2.2 一份 children 分流总结

至少要整理出这几个关键事实：

1. `patchChildren` 第一层按 children 形态分流
2. 文本 children 更新通常直接走文本替换
3. 数组 children 更新会继续进入数组 diff
4. 新 children 为空时需要卸载旧 children
5. 只有数组对数组时才会进入 `patchKeyedChildren`

### 2.3 一份 keyed diff 步骤总结

至少要覆盖下面这些步骤：

1. 头部同步
2. 尾部同步
3. 新增节点
4. 删除节点
5. 中间乱序段映射
6. 最长递增子序列
7. move / mount

要求：

- 能解释为什么先做头尾同步
- 能解释 LIS 解决的是“减少移动”

### 2.4 一份这 5 个问题的答案

至少能回答下面这些问题：

1. 同类型元素更新时，为什么走 `patchElement` 而不是重新 `mountElement`？
2. `patchElement` 为什么把 props 和 children 分开处理？
3. `patchChildren` 第一层按什么维度分流？
4. `patchKeyedChildren` 为什么先做头尾同步？
5. 最长递增子序列在 keyed diff 里到底解决什么问题？

### 2.7 一份验收问答

至少要能独立回答：

1. 为什么同类型元素更新时会优先复用 DOM？
2. 为什么 children diff 要先按文本 / 数组 / 空分流？

4. LIS 为什么能减少 DOM move？
5. 哪些节点会被删除，哪些节点会被移动，哪些节点会被复用？

## 3. 建议产出形式

推荐至少整理成下面 4 份内容：

- 一张元素更新总图
- 一张 children 分流图
- 一张 keyed diff 步骤图
- 一份结构化笔记：元素更新、children 分流、keyed diff、验收问答

## 4. 推荐阅读顺序

1. `patch`
2. `processElement`
3. `mountElement / patchElement`
4. `patchChildren`
5. `patchKeyedChildren`
6. `move / unmount`

## 5. Day04 不纳入产出范围

- `runtime-dom` 的 `patchProp`
- `compiler-core`
- SSR / hydration
- `Transition` / `Teleport` / `Suspense`

- 能解释 `patchChildren` 和 `patchKeyedChildren` 的分工
- 能独立回答本清单中的 5 个验收问题
