# Day07 产出清单

> 根据 `VueSource/docs/vue源码阅读第七天打断点.md` 整理  
> 目标：把“第七天到底要产出什么”收敛成可执行、可验收的清单

## 1. Day07 核心目标

第七天的目标不是继续停留在 `compiler-core` 的抽象骨架，而是把 `compiler-dom` 和 runtime 优化信息亲手接起来。

第七天只要求走通下面这条主链：

`compiler-dom compile(...)`  
`-> parserOptions`  
`-> DOMNodeTransforms / DOMDirectiveTransforms`  
`-> patch flag / helper / block 信息`  
`-> render code`  
`-> runtime 消费 patch flag 与 dynamicChildren`

只要这条链路能通过断点走通，并且能用自己的话讲清楚，Day07 就算完成。

## 2. 必须产出

### 2.1 一份编译分层笔记

至少要整理出下面这条关系：

```text
compiler-core
  -> 通用模板编译能力

compiler-dom
  -> DOM 平台 parserOptions
  -> DOM 平台 transforms
```

要求：

- 说明 `compiler-dom` 为什么不是重新实现一套编译器
- 说明它比 `compiler-core` 多补了什么

### 2.2 一份 DOM transforms 总结

至少要覆盖：

- `transformStyle`
- `transformVHtml`
- `transformVText`
- `transformModel`
- `transformOn`

要求：

- 每个 transform 都要说明“为什么更适合放在 DOM 平台层”

### 2.3 一份 patch flag / block tree 总结

至少要说明：

1. patch flag 来自编译期分析
2. render code 会打印出这些优化信息
3. `openBlock / createElementBlock / createBlock` 会参与 block 建立
4. runtime 会把动态节点收集到 `dynamicChildren`
5. 更新时可以优先 patch 动态节点

### 2.4 一份这 5 个问题的答案

至少能回答：

1. `compiler-dom` 相比 `compiler-core` 多补了什么？
2. 为什么 DOM 平台需要自己的 `parserOptions`？
3. 为什么 DOM 指令转换不适合全放在 `compiler-core`？
4. patch flag 是怎样在编译阶段被分析出来的？
5. block tree / `dynamicChildren` 为什么能让 runtime 更新更快？

### 2.7 一份验收问答

至少要能独立回答：

1. 为什么 `compiler-dom` 必须存在于 `compiler-core` 之上？
